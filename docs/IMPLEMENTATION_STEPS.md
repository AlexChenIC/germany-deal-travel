# Implementation Steps

这是“柏林家庭出行雷达”的执行清单。原则是：每一步尽量小，完成后构建、提交、部署，再进入下一步；除非用户明确调整优先级。

## Step 0: 基线整理

状态：已完成。

目的：确保仓库从干净、可构建的状态继续扩展。

任务：
- 检查 worktree 是否有中断开发留下的半成品。
- 清理 `src/App.tsx` 中未完成的推荐页草稿。
- 运行 `npm run build`。
- 在开始新功能前确认 `git status --short` 没有无关代码改动。

完成标准：
- 构建通过。
- 没有半成品 UI 留在主应用。
- 仓库有清晰的基线提交。

## Step 1: 路线图和执行计划

状态：已完成。

目的：把已经讨论过的优化方向固化为可执行任务，避免后续一次混入太多功能。

任务：
- 扩展 `docs/ROADMAP.md`，整理产品和工程路线图。
- 新增 `docs/IMPLEMENTATION_STEPS.md`，记录具体步骤、任务和完成标准。
- 在 README 中加入开发路线入口，方便后续回看。

完成标准：
- 路线图和执行步骤已经提交。
- 不引入运行时行为变化。

## Step 2: 每日自动检索可视化

状态：已完成。

目的：让已经存在的每日自动更新更明显、更可诊断。

当前状态：
- `.github/workflows/pages.yml` 已经支持 push、手动触发和每日 cron。
- workflow 已经在部署前运行 `npm run update-data`。

任务：
- 在页面上增加自动化状态模块。
- 展示 `radar.generatedAt`、来源成功/失败/跳过数量，以及缓存回退提示。
- 在条目上增加“今日新增”和“本周新增”标签。
- 文档化柏林本地时间下的定时任务触发时间，以及如何手动运行 workflow。
- 决定 cron 是否需要调整到更适合柏林早晨查看的时间。

完成标准：
- 用户不打开 GitHub Actions 也能知道最近一次数据更新是否健康。
- 来源失败和缓存回退可见。
- 构建通过并部署。

## Step 3: 为我推荐页面

状态：已完成。

目的：根据 Alex 家庭情况，自动挑出值得关注的近期活动和远期旅行优惠。

家庭画像：
- 常驻柏林。
- 3 位成年人和 1 位约 11 个月宝宝。
- 重点偏好：避暑、宝宝友好、长辈友好、低折腾、推车方便、短住时优先泳池/SPA、长途优先 BER 出发。

任务：
- 新增 `为我推荐` tab。
- 创建确定性推荐分组：
  - 近期柏林活动。
  - 周边避暑和短住。
  - 远期阳光/全包优惠。
  - 邮轮和大型旅行观察。
  - 儿童活动建议。
- 每条推荐显示推荐原因。
- 复用收藏和排除动作。
- 将推荐逻辑放到可测试的独立模块中。

完成标准：
- 页面可以基于当前静态数据运行。
- 每条推荐都解释为什么出现。
- 收藏/排除会影响推荐结果。
- 构建和浏览器冒烟测试通过。

## Step 4: 避暑短住页面

目的：解决高温周末在柏林 1-2 小时范围内找冷气可靠、适合宝宝和长辈的酒店问题。

任务：
- 新增 `避暑短住` 页面。
- 定义酒店候选结构：
  - 酒店名、位置、交通时间、预订链接。
  - 房间空调证据。
  - 泳池、桑拿、SPA、湖/河边资源。
  - 宝宝适配和风险。
  - 价格区间和检查时间。
  - 评价风险，尤其是房间闷热或空调差。
- 先录入第一批人工核验候选。
- 增加空调、泳池、SPA、预算和距离筛选。

完成标准：
- 页面足以支持一次真实周末避暑决策。
- 证据和不确定性都明确展示。
- 没有实时查价时，不宣称实时可订。

## Step 5: 前端组件拆分

目的：降低后续开发维护成本。

任务：
- 从 `src/App.tsx` 拆出页面组件。
- 抽取卡片、Tab、指标、收藏/排除按钮、儿童地图、来源状态行。
- 抽取 `useStoredIdSet`。
- 将推荐规则移到 `src/lib/recommendations.ts`。
- 按页面/组件整理 CSS。

完成标准：
- 用户可见行为不变。
- 构建通过。
- 后续功能改动的 diff 明显变小。

## Step 6: 数据源适配器重构

目的：让自动化检索更容易扩展，也更不容易因为单个来源失败而影响整体。

任务：
- 拆分 RSS 解析、visitBerlin 解析、来源健康、缓存回退、分类、打分、中文标题生成。
- 为当前来源添加 parser fixtures。
- 定义 source adapter contract。
- 准备接入更多高价值家庭旅行源。

完成标准：
- `scripts/update-data.ts` 更偏向编排层。
- 当前来源输出保持等价。
- 至少覆盖当前来源类型的 parser 测试。

## Step 7: 儿童活动增强

目的：让儿童活动页可以直接帮助日常和周末决策。

任务：
- 增加热天适配、雨天适配、宝宝年龄适配、推车友好、换尿布、室内制冷、核验日期等字段。
- 增加对应筛选。
- 对信息过期条目显示复查提示。
- 补充儿童活动、游泳课、家庭中心、室内活动等来源。

完成标准：
- 可以快速筛选“热天 + 11 个月宝宝 + 室内”。
- 每条资料能看出是否近期核验。

## Step 8: 地图增强

目的：在控制成本和 API key 风险的前提下提高地图实用性。

任务：
- 保留所有地点的 Google Maps 外部跳转链接。
- 保留 Maps Embed API 作为可选单地点内嵌地图。
- 评估 Leaflet/OpenStreetMap 作为免费多点筛选地图。
- 只有在确实需要 Google 原生多点地图体验时，再使用 Google Maps JavaScript API。

完成标准：
- 地图能支持筛选后的多点浏览。
- 没有 API key 时仍有可用回退。

## Step 9: 收藏可迁移

目的：让计划可以跨浏览器、跨设备、跨家庭成员流转。

任务：
- 增加收藏和排除列表 JSON 导出/导入。
- 增加可复制/可分享的 shortlist 文本。
- 后续评估私人云同步。

完成标准：
- 收藏清单可以迁移到另一台设备或分享给家人。

## Step 10: 质量门禁

目的：让项目随着自动化和数据源增长仍然可靠。

任务：
- 增加 lint 和格式化。
- 为推荐规则和分类器增加单元测试。
- 为 parser 增加 fixture 测试。
- 增加 Playwright 冒烟测试，覆盖主页面、收藏、儿童页和地图逻辑。
- 在 GitHub Actions 部署前运行检查。

完成标准：
- CI 能在部署前捕捉常见回归。
- 每日自动化更新更可信。
