# Roadmap

这份路线图把当前的“柏林家庭出行雷达”继续推进成一个真正可长期使用的家庭决策工具：既服务柏林日常生活，也服务柏林周边短住、德国境内旅行、欧洲/周边度假和邮轮/全包产品观察。

## 当前基线

- Vite/React 静态站点，部署在 GitHub Pages。
- GitHub Actions 已经支持每日自动更新公开 RSS/HTML 数据源。
- 卡片优先展示中文标题，同时保留原始标题，方便回源核对。
- 已有“收藏”和“排除”能力，数据保存在本地浏览器。
- 已有柏林儿童活动页面，包含儿童咖啡馆、亲子音乐会、开放活动、博物馆/剧场、游泳课等资料。
- 儿童活动页已有 Google Maps 跳转链接，并支持可选的 Maps Embed API 内嵌地图。
- 已有信息源健康状态和路线规划页面。

## Phase 0: 基线整理

状态：已完成。

- 清理中断开发留下的半成品代码。
- 确认主应用可以从干净基线正常构建。
- 后续功能按小步骤推进，避免多个大改动混在一起。

## Phase 1: 执行计划

状态：已完成。

- 用 `docs/IMPLEMENTATION_STEPS.md` 保存一步步的执行清单。
- 每一步都要足够小，可以单独构建、验证和部署。
- 后续开发优先按这份清单推进，除非用户明确调整优先级。

## Phase 2: 每日自动检索可视化

状态：第一版已完成。

目标：让现有每日自动更新更可信、更可见，也更容易维护。

- 在页面上更突出地展示最后更新时间和数据源健康状态。
- 增加“今日新增”和“本周新增”标识。
- 当某个源失败并回退到缓存时，在页面上更清楚地提示。
- 根据柏林早晨使用习惯，决定 GitHub Actions cron 是否需要调整。
- 后续可考虑通过 GitHub issue、邮件或其他私人渠道发送失败提醒。

## Phase 3: 为我推荐

目标：新增一个“为我推荐”页面，自动挑出最值得 Alex 家庭关注的近期活动和远期旅行优惠。

- 固化当前家庭画像：柏林生活、3 位成年人、约 11 个月宝宝、兼顾长辈、避暑、低折腾、亲子友好。
- 推荐分组包括：近期柏林活动、周边避暑/短住、远期阳光/全包旅行、邮轮和大型旅行、儿童活动。
- 每条推荐都显示推荐原因。
- 复用收藏和排除操作。
- 第一版先使用确定性规则，后续再考虑 AI 摘要或更复杂的排序。

## Phase 4: 避暑短住

目标：覆盖夏季高温周末的真实需求：在柏林 1-2 小时范围内找有可靠房间空调、泳池/SPA、适合宝宝和长辈的酒店。

- 新增“避暑短住”页面。
- 定义酒店候选数据结构：名称、位置、交通时间、预订链接、房间空调证据、泳池、桑拿/SPA、亲子适配、价格区间、检查时间、评价风险。
- 先用人工/半自动方式维护第一批高质量候选。
- 后续再为 Booking、Travelcircus、酒店官网等平台添加价格/可订性检查适配器。

## Phase 5: 前端组件拆分

目标：在功能继续增多前降低维护成本。

- 将 `src/App.tsx` 拆分为页面组件。
- 抽取卡片、Tab、指标、收藏/排除操作、儿童地图、来源状态等公共组件。
- 抽取本地存储集合逻辑。
- 把推荐规则移动到 `src/lib/recommendations.ts`。
- 拆分时保持用户可见行为不变。

## Phase 6: 数据源适配器架构

目标：让新增旅行源、活动源和儿童活动源更安全、更快。

- 拆分 `scripts/update-data.ts` 中的 RSS 解析、visitBerlin 解析、源健康、缓存回退、分类、打分和中文标题生成。
- 为当前源添加 fixture 测试。
- 定义统一 source adapter contract。
- 准备接入 Center Parcs、Travelcircus、thermal spa hotel、Groupon、Eventbrite、Fever、Familotel 等高价值来源。

## Phase 7: 儿童活动增强

目标：把儿童活动页从资料目录升级为可直接做周末/平日计划的工具。

- 为每个地点/活动增加宝宝年龄适配、热天适配、雨天适配、推车友好、换尿布、室内制冷、信息核验日期等字段。
- 添加“11 个月宝宝”“热天”“雨天”“免费/低价”等筛选。
- 对信息过期的条目显示复查提示。

## Phase 8: 地图增强

目标：提升地图浏览能力，同时控制 API 成本和 key 暴露风险。

- 保留所有地点的 Google Maps 外部跳转链接。
- 保留 Maps Embed API 作为可选的单地点内嵌地图。
- 评估 Leaflet/OpenStreetMap 作为免费多点筛选地图。
- 只有当 Google 原生多点地图体验明显值得时，再考虑 Google Maps JavaScript API。

## Phase 9: 收藏可迁移

目标：让家庭计划可以跨浏览器、跨设备转移。

- 增加收藏和排除列表的 JSON 导出/导入。
- 增加可复制的 shortlist 文本摘要。
- 后续评估 GitHub Gist、Cloudflare D1、Supabase 等私人同步方案。

## Phase 10: 质量门禁

目标：随着自动化和来源变多，保持项目可靠。

- 添加 lint 和格式化。
- 为推荐规则和分类器增加单元测试。
- 为数据源解析器增加 fixture 测试。
- 增加 Playwright 冒烟测试，覆盖主要页面、收藏、儿童页和地图回退/内嵌逻辑。
- 在 GitHub Actions 部署前运行这些检查。
